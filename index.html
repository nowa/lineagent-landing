What is Lineage?#

Lineage 是什么？#

Lineage とは？#

Lineage is a continuously-evolving multi-agent runtime built around four primitives. The core philosophy: 谱系一等公民 — every Agent has a parent_id, a generation, a strain_id, a fitness_history, an immutable genome record, and an auditable execution chain.

Lineage 是一个持续演化的多 Agent 运行时，围绕四个核心原语构建。 核心哲学：谱系一等公民 —— 每个 Agent 都有 parent_id、generation、 strain_id、fitness_history、 一份不可变的 genome 记录、以及一条可审计的执行链。

Lineage は 4 つのプリミティブを中心に構築された、継続的に進化する マルチエージェント・ランタイムです。コアとなる哲学は 谱系一等公民（系譜は第一級市民）—— すべての Agent が parent_id、generation、 strain_id、fitness_history、 不変な genome レコード、そして監査可能な実行チェーンを持ちます。

Strain

A domain or task family — email, coding, browser-ops. Defines the eval contract and the boundary of which tools an Agent in this family is allowed to use.

领域或任务族 —— 邮件、编码、浏览器自动化。 定义评估契约，以及该族 Agent 可用工具的边界。

ドメインまたはタスク・ファミリ —— メール、コーディング、ブラウザ操作など。 評価契約と、そのファミリの Agent が使えるツールの境界を定義します。

Agent

An individual instance within a Strain. Carries its own immutable genome snapshot, parent reference, generation number, and fitness history; runs through the 4-state lifecycle candidate → stable → deprecated ⇄ stable → extinct. Multiple Agents can coexist within one Strain — each with its own evolution path.

某个 Strain 下的具体个体。带着自己的不可变 genome 快照、parent 引用、世代号和 fitness 历史，按 4 状态生命周期运行 （candidate → stable → deprecated ⇄ stable → extinct）。 同一个 Strain 下可以并存多个 Agent，各自走各自的演化路径。

ある Strain に属する個体です。自身の不変な genome スナップショット、 親（parent）への参照、世代番号、fitness 履歴を持ち、4 状態のライフサイクル （candidate → stable → deprecated ⇄ stable → extinct）を辿ります。 同じ Strain の中に複数の Agent が共存し、それぞれ独自の進化経路を進みます。

Genome

An immutable JSON capturing prompts, planner config, tool-policy weights. Editing a genome creates a new Agent (a child) — never mutates the parent. Lineage is auditable and replayable because of this.

不可变的 JSON，记录 prompt、planner 配置、tool-policy 权重等。 编辑 genome 会产生一个新 Agent（子代），永远不修改父代。 系谱因此可审计、可回放。

プロンプト、planner 設定、tool-policy の重みなどを記録した 不変な JSON。genome を編集すると新しい Agent（子）が生成され、 親は決して変更されません。これにより系譜は監査可能かつ再生可能になります。

Evolution

CUSUM-triggered candidate generation, offline evaluation, Inbox approval, then a Blue-Green swap with a 7-day or 20-session observation window. Auto-rollback on regression.

CUSUM 触发候选生成 → 离线评估 → Inbox 审批 → Blue-Green 切换 → 7 天或 20 会话观察窗 → 检测到回归自动回滚。

CUSUM トリガによる候補生成 → オフライン評価 → Inbox 承認 → Blue-Green 切替 → 7 日または 20 セッションの観察窓 → リグレッション検出時の自動ロールバック。

The runtime is organized into four subsystems. User input enters through Control, executes inside the AgentKernel, draws on Capability for tools and skills, and is observed by the Evolution loop:

运行时分为四个子系统。用户输入从 Control 进来，在 AgentKernel 里执行， 向 Capability 取工具和技能，由 Evolution 循环观察：

ランタイムは 4 つのサブシステムで構成されます。ユーザー入力は Control から入り、 AgentKernel で実行され、Capability からツールとスキルを取得し、 Evolution ループによって観察されます： 

             user · IM · API · CLI
                       │
                       ▼
         ┌──────────────────────────────┐
         │           Control            │
         │  Gateway · Adapters · Auth   │
         └───────────────┬──────────────┘
                         ▼
         ┌──────────────────────────────┐
         │          Execution           │
         │  AgentKernel · Compose       │
         │  PEO loop · Streaming        │
         └───────┬──────────────────┬───┘
                 ▼                  ▼
    ┌──────────────────────┐  ┌─────────────────────┐
    │      Capability      │  │      Evolution      │
    │   Tools · Skills     │  │   Eval · CUSUM      │
    │   Browser · Memory   │  │   Blue-Green swap   │
    └──────────────────────┘  └─────────────────────┘

Files are the Source of Truth; the SQLite DB is a materialized index that can be rebuilt from data/ at any time.

文件是 Source of Truth；SQLite DB 只是物化索引， 任何时候都可以从 data/ 重建。

ファイルが Source of Truth で、SQLite DB はマテリアライズドなインデックスであり、 いつでも data/ から再構築できます。

Quick start#

快速开始#

クイックスタート#

Lineage is a TUI-first product. Drop into a project directory and run lineage chat:

Lineage 是 TUI 优先的产品。进到一个项目目录，跑 lineage chat：

Lineage は TUI ファーストのプロダクトです。プロジェクトディレクトリに移動して lineage chat を実行します： 

$ cd ~/your-project
$ lineage chat
∴ lineage 0.117.x — pal ready

> review the auth middleware for timing-attack risks

  ● filesystem.read    crates/auth/src/middleware.rs
  ● filesystem.grep    "compare|hmac|verify"  crates/auth/

  Found one — line 47 compares HMAC tags with `==`, which short-
  circuits on first byte mismatch. A sufficiently noisy attacker
  can recover the tag byte-by-byte from response timing.

  Constant-time compare via `subtle::ConstantTimeEq` would fix it.
  Want me to write the patch and run the test suite?

> yes, patch it

  ● filesystem.edit    crates/auth/src/middleware.rs   (1 hunk)
  ● cli.execute        cargo test -p auth --lib
                       passed  17/17  in  3.4s

  Done. Diff is ready to commit.

The first run walks you through provider setup (Anthropic / OpenAI / OpenAI-compat / Codex subscription), asks you to trust the project directory, and then drops into the three-pane chat layout — conversation in the middle, status bar at the bottom, optional context pane on the side.

首次运行会引导你完成 provider 配置 （Anthropic / OpenAI / OpenAI-compat / Codex 订阅）， 询问是否信任当前项目目录，然后进入三栏聊天布局 —— 中间是会话，底部是状态栏，侧边可选上下文面板。

初回起動時はプロバイダ設定 （Anthropic / OpenAI / OpenAI 互換 / Codex サブスクリプション）の ウィザードに誘導され、プロジェクトディレクトリを信頼するか確認したのち、 3 ペインのチャットレイアウトに入ります —— 中央が会話、下部がステータスバー、サイドにコンテキストペイン（任意）。

Sessions are persistent. Run lineage chat --resume to pick up where you left off. Run lineage agents to list lineage trees, lineage cost to see token spend, lineage soul to edit the active strain's personality.

会话是持久化的。lineage chat --resume 接续上次对话。 lineage agents 列出系谱树； lineage cost 查看 token 花费； lineage soul 编辑当前 strain 的人格设定。

セッションは永続化されます。lineage chat --resume で続きから再開できます。 lineage agents で系譜ツリーの一覧、 lineage cost でトークン使用量、 lineage soul で現在の strain のパーソナリティ編集ができます。

内置 Strain

内蔵 Strain

Lineage ships with six built-in Strains. Each has an immutable seed genome you can fork to create your own variants. Three are user-facing; three live inside the evolution pipeline.

Lineage 出厂内置六个 Strain。每个都有一份不可变的 seed genome， 你可以 fork 它来派生自己的变体。其中三个面向用户，三个跑在演化管线内部。

Lineage には 6 つの Strain が組み込まれています。それぞれに不変な seed genome があり、fork して独自のバリアントを作れます。 ユーザー向けが 3 つ、進化パイプライン内部で動くのが 3 つです。

User-facing

面向用户

ユーザー向け

pal
Your primary agent. Every request flows through pal first — it handles tasks directly when it can, and delegates to specialists when domain expertise is needed.
你的主 Agent。所有请求先到 pal —— 能自己处理就自己处理， 需要领域专长时再委派给专家。

Evolution pipeline

演化管线

進化パイプライン

analyzer
Diagnostic engine. Examines evaluation results to find patterns aggregate scores hide, attributes failures to root causes, and decides which mutation operators the evolution pipeline applies next.
诊断引擎。检查评估结果，找出聚合分数遮蔽的模式， 把失败归因到根因，决定演化管线下一步用哪个 mutation operator。
診断エンジン。評価結果を検査して集約スコアでは見えない パターンを見つけ、失敗を根本原因に帰属させ、進化パイプラインが 次にどの mutation operator を適用するかを決定します。
grader
Evaluation oracle. Scores Agent outputs against EvalContract dimensions with cited evidence. Drives evolution decisions — accuracy is everything.
评估神谕。按 EvalContract 维度给 Agent 输出打分， 必须提供证据。驱动演化决策 —— 准确性高于一切。
評価のオラクル。EvalContract の各次元に対して Agent の 出力を採点し、根拠を必ず示します。進化の意思決定を駆動 —— 正確性がすべてです。
comparator
Blind judge. Compares two Agent outputs WITHOUT knowing which is the champion and which is the candidate. Bias toward either side corrupts the entire pipeline.
盲审法官。比较两个 Agent 的输出， 不知道哪个是 champion、哪个是 candidate。 任何一侧的偏见都会污染整条管线。
ブラインド・ジャッジ。2 つの Agent 出力を比較しますが、 どちらがチャンピオンでどちらが候補かは知りません。 どちらかへのバイアスはパイプライン全体を汚染します。

Tools#

Tools#

Tools#

Tools are how Agents act on the world. Lineage ships a Rust-native toolbox covering the moves a coding-and-ops agent actually needs: read & edit files, run shell, drive a browser, fetch web pages, delegate to other Agents, schedule background work. Every tool is gated by per-Strain genome weights — a Strain that doesn't declare a tool simply can't call it. No external runtime, no Node sidecar, no MCP-only fallback.

Tools 是 Agent 作用于世界的手段。Lineage 自带一套 Rust 原生工具箱， 覆盖一个 coding/ops Agent 真正需要的动作：读写文件、跑 shell、操控浏览器、 抓网页、委派给其他 Agent、调度后台任务。每一个 tool 都按 Strain 的 genome 权重门控 —— Strain 没声明的 tool 就调不到。不需要外部运行时、不需要 Node sidecar，不强制走 MCP fallback。

Tools は Agent が世界に働きかける手段です。Lineage には Rust ネイティブの ツールボックスが同梱され、コーディング/オプス用エージェントに必要な動作を カバーします —— ファイルの読み書き、シェル実行、ブラウザ制御、Web 取得、 他 Agent への委譲、バックグラウンドジョブのスケジュール。すべてのツールは Strain の genome 重みでゲートされ、宣言していない Strain は呼び出せません。 外部ランタイムも Node サイドカーも MCP 必須のフォールバックもありません。

シェルコマンドを実行。Strain ごとの allowlist（正規表現）、 実行ごとのサンドボックス階層選択、書き込み系コマンドには任意で human-in-the-loop 確認。

Strain 范围内的 JSONL 记忆：save · recall · forget。 Project memory 按 project_key 分桶，跟跨项目的 strain memory 分开。
Strain スコープの JSONL メモリ：save · recall · forget。プロジェクトメモリは project_key 単位で、クロスプロジェクトの strain メモリとは分離されます。

Agent Kernel#

Agent Kernel#

Agent Kernel#

The Agent Kernel is the part of Lineage that turns a chat turn into a stream of tool calls and tokens. It is a pure-computation core (Ports & Adapters) — no I/O, no database, no provider coupling — wrapped by the runtime that does. Five capabilities are baked in; together they're what makes Lineage hold up across long sessions.

Agent Kernel 是 Lineage 把"一轮对话"转成"工具调用流 + token 流"的那块。 它是纯计算核心（Ports & Adapters 模式）—— 不做 I/O、不碰 DB、不绑定 provider —— 外面包着真正做 I/O 的 runtime。它内置五项能力，组合起来让 Lineage 能在长会话里撑住：

Agent Kernel は Lineage が「1 ターンの対話」を「ツール呼び出しストリーム + トークンストリーム」に変換するコアです。純粋計算コア（Ports & Adapters）—— I/O なし、DB アクセスなし、プロバイダ結合なし —— で、I/O を担うランタイムが 周囲を取り囲みます。次の 5 つの能力が組み込まれており、長セッションでの 持続性を支えます：

Context engineering
上下文工程（Context engineering）
コンテキストエンジニアリング
Every prompt is built by a deterministic PromptCompiler — system prompt, project profile, memory recall, conversation history, current tool state, all assembled with explicit budget caps. Conversations are JSONL-persisted (data/conversations/{id}/active.jsonl), so a crash or restart resumes mid-turn instead of losing context.
每个 prompt 都由确定性的 PromptCompiler 编译 —— system prompt、项目 profile、memory recall、会话历史、当前工具状态， 按显式预算上限组装好。会话用 JSONL 持久化 （data/conversations/{id}/active.jsonl）， 崩溃或重启不会丢上下文，可以从断点续上。
すべてのプロンプトは決定論的な PromptCompiler によって組み立てられます —— system プロンプト、プロジェクトプロファイル、メモリリコール、会話履歴、 現在のツール状態を明示的な予算上限で組み合わせます。 会話は JSONL で永続化され （data/conversations/{id}/active.jsonl）、 クラッシュや再起動でも途中から再開でき、コンテキストは失われません。
Compaction
Compaction（压缩）
コンパクション（圧縮）
Three-layer trace storage. Level 0 (always: metadata + tool names) and Level 1 (always: input + output preview) are unconditional. Level 2 (full payloads) is sampled only on success for regression mining. When the live conversation approaches the model's context window, the kernel compacts older turns into a structured summary; raw history stays on disk for replay.
三层 trace 存储。Level 0（始终：元数据 + 工具名）和 Level 1（始终：input + output 预览）都是无条件记录的； Level 2（完整 payload）只在成功时采样，用于回归挖掘。 活跃会话接近模型 context window 时，kernel 把更早的 turn 压成结构化摘要 —— 原始历史还在磁盘上，可回放。
3 層のトレース保存。Level 0（常時：メタデータ + ツール名）と Level 1（常時：入出力プレビュー）は無条件で記録され、 Level 2（完全ペイロード）は成功時のみサンプリングされ、 リグレッション分析に使われます。アクティブ会話がモデルの context window に近づくと、kernel は古いターンを構造化サマリへ圧縮します —— 生履歴はディスクに残り、再生可能です。
Memory (4 layers)
Memory（四层）
メモリ（4 層）
Conversation (this turn's working set) → Project (per project_key: data/projects/{key}/memory.jsonl) → Strain (cross-project domain knowledge: data/memory/{strain_id}.jsonl) → Genome (immutable, lifts only via evolution). Higher layers don't pollute lower ones; lower-layer write rate is bounded.
Conversation（这轮 working set）→ Project（按 project_key： data/projects/{key}/memory.jsonl）→ Strain（跨项目的领域知识： data/memory/{strain_id}.jsonl）→ Genome（不可变，只通过演化路径下沉）。 高层不污染低层；低层写入速率有上限。
Conversation（このターンのワーキングセット）→ Project（project_key 単位： data/projects/{key}/memory.jsonl）→ Strain（クロスプロジェクトのドメイン知識： data/memory/{strain_id}.jsonl）→ Genome（不変、進化経由でのみ下層に降りる）。 上位層が下位層を汚染することはなく、下位層の書き込みレートには上限があります。
Long-running tasks
长任务（Long-running tasks）
長時間タスク
Multi-step plans live as Task → Step → Job (ADR-053). Task state is file-first (data/tasks/{id}/task.json), each Step is its own compose_turn with its own conversation, and a background watchdog raises Steps that haven't progressed in N minutes — so plans don't quietly die in the middle.
多步骤计划走 Task → Step → Job 三层（ADR-053）。 Task 状态 file-first（data/tasks/{id}/task.json）， 每个 Step 是一次独立的 compose_turn，自带会话； 后台 watchdog 把 N 分钟没进展的 Step 拎出来 —— 计划不会在中间静默地死掉。
マルチステップ計画は Task → Step → Job として実行されます（ADR-053）。Task 状態は file-first （data/tasks/{id}/task.json）、各 Step は独立した compose_turn（独自の会話）、N 分間進捗のない Step は バックグラウンド watchdog に拾われます —— 計画が途中で静かに死ぬことはありません。
Prompt cache
Prompt cache（提示缓存）
プロンプトキャッシュ
Lineage uses provider-side caching wherever available — Anthropic cache_control, OpenAI cached input, DeepSeek cached prompt prefix. The kernel structures the prompt prefix to be cache-friendly (system + project profile pinned, memory near the bottom), so repeat turns of a long session land at cache-hit pricing instead of full prefill.
能命中 provider cache 的就命中 —— Anthropic cache_control、OpenAI cached input、 DeepSeek cached prompt prefix。 kernel 在编译 prompt 时就把前缀做成 cache-friendly 的形状 （system + 项目 profile 固定在前，memory 拉到后面）， 长会话的重复轮次按 cache-hit 价格走，不是 full prefill。
利用可能な場面ではプロバイダ側キャッシュを使います —— Anthropic cache_control、OpenAI cached input、 DeepSeek cached prompt prefix。kernel はプロンプト前置部を キャッシュフレンドリな形に構成し（system + プロジェクトプロファイルを 前段に固定、memory は後段）、長セッションの繰り返しターンは full prefill ではなく cache-hit 価格で実行されます。
Native advisor pattern
原生 advisor 模式
ネイティブ advisor パターン
Per-provider advisors translate the kernel's provider-neutral request into provider-best-practice calls — Anthropic gets thinking blocks + cache_control + tool_use streaming; OpenAI Responses API gets reasoning + structured outputs; Codex subscription gets native attribution headers. The same Agent runs on any of nine providers without rewrites.
每个 provider 都有专门的 advisor，把 kernel 的 provider-neutral 请求翻译成各家的 best practice —— Anthropic 用 thinking block + cache_control + tool_use 流式； OpenAI Responses API 用 reasoning + structured outputs； Codex 订阅带 native attribution header。 同一个 Agent 跑在 9 家 provider 上不用改写。
プロバイダごとに advisor があり、kernel のプロバイダ非依存 リクエストを各社のベストプラクティス呼び出しに翻訳します —— Anthropic は thinking block + cache_control + tool_use ストリーミング、 OpenAI Responses API は reasoning + structured outputs、Codex サブスクリプションはネイティブ帰属ヘッダ付き。同一 Agent が 9 つのプロバイダのどれでも書き換えなしに動きます。

Where this lands competitively: most coding-agent products give you context engineering only inside their own UI, route a single provider, and stop the trail at the conversation boundary. Lineage stores every turn as a replayable artifact, evolves the Agent that ran it based on those replays, and lets you swap the underlying model the next morning. The kernel is what makes that cross-session, cross-provider continuity possible.

在竞品里的位置：大多数 coding agent 产品只在自家 UI 里给你 context engineering，只对接一家 provider，对话边界一过线索就断。 Lineage 把每轮 turn 存成可回放的 artifact，基于这些回放演化跑出它的 Agent， 第二天还允许你换底模。kernel 是让"跨会话、跨 provider 的连续性"成立的那块。

競合との位置取り：大半のコーディング・エージェント製品はコンテキスト エンジニアリングを自社 UI 内に閉じ込め、単一プロバイダにルーティングし、 会話境界で痕跡が途切れます。Lineage はすべてのターンを再生可能な アーティファクトとして保存し、その再生に基づいて Agent を進化させ、 翌日には基盤モデルを差し替えることもできます。kernel は 「セッション越え・プロバイダ越えの連続性」を成立させる中核です。

自動更新#

lineage self-update is channel-aware. If you installed via Homebrew, it redirects you to brew upgrade lineage; npm to npm update -g; Cargo to cargo install lineage --force; snap to sudo snap refresh. Only direct (install.sh) installs do an in-place atomic POSIX rename swap, with a .prev hardlink backup.

lineage self-update 是频道感知的。Homebrew 装的会被引导去 brew upgrade lineage；npm 装的引导到 npm update -g；Cargo 装的引导到 cargo install lineage --force；snap 装的引导到 sudo snap refresh。 只有直装（install.sh）会就地原子 POSIX rename 替换， 并带 .prev hardlink 备份。

lineage self-update はチャンネルを認識します。 Homebrew 経由でインストールされた場合は brew upgrade lineage に、 npm の場合は npm update -g、Cargo は cargo install lineage --force、snap は sudo snap refresh にリダイレクトします。 直接インストール（install.sh）の場合のみ、その場で POSIX rename によるアトミックな入れ替えを行い、 .prev ハードリンクのバックアップを残します。

后台自动更新是主动 opt-in 的。默认情况下后台只做通知 —— 每 30 分钟查一次 GitHub，发现新版本就在启动时打条 banner，仅此而已。 要打开静默安装，编辑 ~/.config/lineage/lineage.toml：

バックグラウンド更新はオプトインです。デフォルトでは通知のみ —— 30 分ごとに GitHub をチェックし、新バージョンがあれば起動時にバナーを出すだけです。 サイレントインストールを有効にするには、 ~/.config/lineage/lineage.toml を編集してください：

Telemetry & privacy#

遥测与隐私#

テレメトリとプライバシー#

Lineage ships anonymous usage telemetry default-on. The data helps prioritize features and catch real regressions. We deliberately skipped the first-run consent prompt — every other consent dialog you've ever seen in a CLI tool taught you to tap "no" on reflex, which means default-off telemetry measures only a self-selected sub-population and biases every product decision. Default-on with a kill-switch gives us a representative sample while keeping you in control.

Lineage 默认开启匿名使用遥测。这些数据帮我们优先排功能，也帮我们发现真正的回归。 我们刻意没做首启同意对话框 —— 你在每个 CLI 工具里见过的每个"是否同意收集数据" 弹窗都在训练你下意识按"否"，这意味着默认 off 的遥测只能采到一群自选样本， 每个产品决策都被偏置数据带跑偏。默认 on + 一键关停才能让我们看到代表性样本， 同时把控制权留给你。

Lineage は匿名の使用テレメトリをデフォルトで有効にして配布します。 このデータは機能の優先順位付けと、本当のリグレッションの早期発見に役立ちます。 初回同意ダイアログを意図的に排除しました —— あなたがこれまで CLI ツールで目にしてきた同意ダイアログは、 すべて反射的に「いいえ」を押すよう訓練してしまっています。 つまりデフォルト off のテレメトリは自己選択された部分集団のみを測定し、 あらゆる製品判断を歪めます。デフォルト on + キルスイッチであれば、 代表的なサンプルを得つつ、コントロールはあなたの手元に残ります。

Why Rust?
为什么用 Rust？
なぜ Rust？
The Control / Execution / Capability planes need predictable latency and zero GC pauses on the hot path. Tooling is mature enough that we get these for free without paying in productivity. The TUI specifically benefits — Ratatui is extremely fast, and the streaming-tokens-into-three-panes rendering would be hard to keep tear-free in a GC'd runtime.
Control / Execution / Capability 三个平面需要可预测的延迟， 热路径不能有 GC 停顿。Rust 的工具链够成熟，这些好处不需要在生产力上付代价就能拿到。 TUI 尤其受益 —— Ratatui 极快，把流式 token 实时渲染到三个面板， 在带 GC 的运行时里很难做到不撕裂。
Control / Execution / Capability の各プレーンには予測可能なレイテンシと、 ホットパスにおける GC ポーズゼロが必要です。 Rust のツーリングは十分に成熟しており、生産性の犠牲なしにこれらが手に入ります。 特に TUI が恩恵を受けます —— Ratatui は非常に高速で、 トークンを 3 ペインにストリーミング表示するレンダリングを GC 付きランタイムで tear-free に保つのは難しいでしょう。
Where does my data live?
我的数据存在哪里？
データはどこに保存されますか？
On your machine. Conversation history is JSONL under data/conversations/. Genomes and strains are YAML/JSON under data/. Memory is JSONL under data/memory/. The SQLite DB is purely an index and can be rebuilt from these files via lineage rebuild-index at any time. Files are the Source of Truth.
在你自己的机器上。会话历史是 data/conversations/ 下的 JSONL； Genome 和 Strain 是 data/ 下的 YAML/JSON； 记忆是 data/memory/ 下的 JSONL。 SQLite DB 只是索引，任何时候都可以 lineage rebuild-index 重建。文件是 Source of Truth。
あなたのマシン上です。会話履歴は data/conversations/ 下の JSONL、 Genome と Strain は data/ 下の YAML/JSON、 メモリは data/memory/ 下の JSONL に保存されます。 SQLite DB は純粋にインデックスであり、いつでも lineage rebuild-index で再構築可能です。 ファイルが Source of Truth です。
Is the agent autonomous? What about safety?
Agent 是自主运行的吗？安全性怎么保证？
Agent は自律的に動作しますか？安全性は？
The Capability Plane runs in a tiered sandbox — Wasm/WASI in MVP, Linux namespace + seccomp + cgroups in Phase 1, MicroVM in Phase 2. Tool policy is per-strain via genome weights: a strain that doesn't list cli_execute in its allowlist literally cannot shell out. Browser automation goes through a dedicated Chromium profile and binds loopback-only.
Capability Plane 跑在分层沙箱里 —— MVP 用 Wasm/WASI，Phase 1 用 Linux namespace + seccomp + cgroups， Phase 2 用 MicroVM。 工具策略是按 strain 走 genome 权重的：一个 strain 如果 allowlist 里没列 cli_execute，它字面上就 shell 不出去。 浏览器自动化走独立的 Chromium profile，且只绑 loopback。
Capability プレーンは階層化されたサンドボックスで実行されます —— MVP は Wasm/WASI、Phase 1 は Linux namespace + seccomp + cgroups、 Phase 2 は MicroVM。 ツールポリシーは strain ごとに genome の重みで管理されます —— allowlist に cli_execute がない strain は文字通りシェルを呼べません。 ブラウザ自動化は専用の Chromium プロファイルを使い、ループバックのみにバインドします。
What's a "genome"?
"Genome" 是什么？
「genome」とは？
An immutable JSON snapshot of an Agent's prompts, planner config, and tool-policy weights. Genome is content-addressed by SHA-256; editing one creates a new Agent (a child) and the old one stays archived for replay. This makes the lineage auditable: you can always answer "what changed?" by diffing parent vs child genomes.
某个 Agent 的 prompt、planner 配置、tool-policy 权重的不可变 JSON 快照。 Genome 用 SHA-256 内容寻址；编辑一个 genome 会产生新 Agent（子代）， 旧的归档保留以供回放。这让系谱可审计 —— "什么变了？"永远可以通过 diff 父代 vs 子代 genome 答出来。
ある Agent のプロンプト、planner 設定、tool-policy の重みの 不変 JSON スナップショットです。Genome は SHA-256 によるコンテンツアドレシング —— 編集すると新しい Agent（子）が生成され、古い genome は再生用に保管されます。 これにより系譜は監査可能になります —— 「何が変わったか？」は親と子の genome を diff すれば常に答えられます。
Linux / macOS only?
只支持 Linux / macOS？
Linux / macOS のみ？

Why not just use Claude Code / Cursor / Copilot directly?
为什么不直接用 Claude Code / Cursor / Copilot？
Claude Code / Cursor / Copilot をそのまま使うのではダメ？
You can, and you should — they're great. Lineage is for when you want the agents themselves to evolve over time based on evaluation. Claude Code is a tool; a Lineage Agent is a lineage node with a fitness history. If your only need is "edit this file, fix this bug, write this test," you don't need Lineage. If you want to tune which agent personalities succeed at which tasks across weeks, and have that selection happen mechanically rather than by gut, that's where Lineage starts to pay off.
完全可以，而且也应该用 —— 它们都很好用。 Lineage 适合的是另一个需求：希望 Agent 本身随时间基于评估演化。 Claude Code 是一个工具；Lineage 的 Agent 是一个有 fitness 历史的系谱节点。 如果你的需求就是"改这个文件、修这个 bug、写这个测试"， 你不需要 Lineage。 如果你想调试"哪种 Agent 人格在哪类任务上跨周稳定胜出"，并且想让这个筛选机械化、 不靠直觉，那才是 Lineage 开始发挥价值的地方。
使ってください —— これらは優秀なツールです。Lineage が向くのは別のニーズ —— Agent そのものを評価に基づいて時間とともに進化させたい場合です。 Claude Code はツールであり、Lineage の Agent は fitness 履歴を持つ系譜ノードです。 必要なのが「このファイルを編集、このバグを修正、このテストを書く」だけなら、 Lineage は不要です。「どのエージェント人格がどんなタスクで週単位で安定して勝つか」を 機械的に（直感ではなく）選別したい場合、そこから Lineage が真価を発揮します。
